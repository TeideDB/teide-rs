SQL/PGQ — Graph Queries in Teide

Extending Teide with ISO SQL/PGQ (SQL:2023 Part 16) graph pattern matching — property graphs over existing tables, GRAPH_TABLE queries, and full composability with SQL analytics.

Context: Why Graph?

Graph databases like FalkorDB and Neo4j have proven the value of property graph queries for knowledge graphs, social networks, fraud detection, and AI/RAG pipelines. FalkorDB uses sparse matrix algebra (via GraphBLAS); Neo4j uses its own Cypher language.

The industry is converging on SQL/PGQ (ISO 9075-16:2023) — graph pattern matching integrated directly into SQL. Oracle 23ai, Postgres 18, and DuckDB (via DuckPGQ) are all adopting this standard. It gives you Cypher-like pattern matching without abandoning SQL's strengths in analytics.

Teide already has a strong foundation: CSR adjacency indexes, 4 graph traversal operations, and a graph-aware optimizer — all in the C engine. The gap is purely at the query language level.

What Teide Already Has

ENGINE

CSR Adjacency Indexes

Double-indexed (forward + reverse) compressed sparse row format. O(1) neighbor degree lookup, O(1) range access. Save/load/mmap persistence.

ENGINE

4 Graph Operations

EXPAND (1-hop), VAR_EXPAND (BFS), SHORTEST_PATH, and WCO_JOIN (Leapfrog Triejoin for cyclic patterns like triangle detection).

OPTIMIZER

Graph-Aware Optimization

Sideways Information Passing (SIP) propagates filters backward through traversals. Factorized vectors avoid materializing cross-products.

EXECUTOR

Unified Relational + Graph

Graph ops compose with filter, join, aggregate, sort, window functions in the same DAG. One optimizer, one executor.

Key insight: The engine-level primitives are already competitive. The main gap is accessibility — graph operations can only be used via the Rust API, not through any query language.

Why SQL/PGQ?

We considered three approaches for exposing graph queries:

ApproachGraph QueriesAnalyticsEcosystemStandard
Cypher only Excellent Poor (no GROUP BY, window functions) Bolt protocol, limited tooling De facto (Neo4j)
Datalog Excellent (recursive rules) Poor (needs non-standard extensions) Almost none Academic
SQL/PGQ Excellent (Cypher-like MATCH) Excellent (full SQL) PgWire, every SQL tool ISO SQL:2023

SQL/PGQ gives us Cypher-like pattern matching inside SQL. Graph results flow directly into GROUP BY, ORDER BY, JOIN, window functions — no impedance mismatch. Every Postgres client (DBeaver, psql, Grafana, psycopg2) works immediately via PgWire.

Oracle 23ai, Postgres 18, and DuckDB (DuckPGQ) are all adopting this standard. We join that wave rather than inventing something new.

Architectural Advantage

Teide's CSR + morsel-driven columnar execution is more practical than pure linear algebra for many workloads:

  • CSR gives O(1) neighbor access — no matrix multiply needed for selective queries
  • Columnar engine handles analytics natively — most graph databases struggle with aggregation, joins, and window functions
  • Unified optimizer — graph traversals and relational operations share one optimization pipeline (SIP, predicate pushdown, fusion)
  • No impedance mismatch — graph results flow directly into GROUP BY, ORDER BY, JOIN without serialization boundaries

Most graph databases can't do efficient analytics. Most analytical databases can't do graph traversals. Teide does both in one engine.

Proposed Architecture

SQL/PGQ Query | +------v-------+ | SQL Parser | sqlparser crate (DuckDB dialect) | (extended) | + GRAPH_TABLE / MATCH parsing +------+-------+ | +------------+-------------+ | | | Relational GRAPH_TABLE CREATE AST + MATCH PROPERTY GRAPH | | | +------------+-------------+ | +------v-------+ | PGQ Planner | MATCH pattern → expand / var_expand | (new) | / shortest_path / wco_join ops +------+-------+ | +-------v--------+ | Teide Graph | Lazy DAG with graph ops | (existing) | +-------+--------+ | +-------v--------+ | Optimizer | SIP, pushdown, fusion, DCE | (existing) | +-------+--------+ | +-------v--------+ | Executor | Morsel-driven parallel | (existing) | +----------------+

Changes are isolated to the parser and a new PGQ planner module. The optimizer and executor remain untouched.

Phase 1: Property Graphs + Basic MATCH

FOUNDATION

Define property graphs over existing tables and run basic 1-hop pattern matching.

CREATE PROPERTY GRAPH

Following the SQL:2023 standard, a property graph is a named schema object that maps existing tables to graph nodes and edges:

-- Load relational data as usual
CREATE TABLE persons AS SELECT * FROM read_csv('persons.csv');
CREATE TABLE companies AS SELECT * FROM read_csv('companies.csv');
CREATE TABLE knows_edges AS SELECT * FROM read_csv('knows.csv');
CREATE TABLE works_at AS SELECT * FROM read_csv('works_at.csv');

-- Define a property graph over those tables
CREATE PROPERTY GRAPH social
VERTEX TABLES (
  persons  LABEL Person,
  companies LABEL Company
)
EDGE TABLES (
  knows_edges
    SOURCE KEY (person1_id) REFERENCES persons (id)
    DESTINATION KEY (person2_id) REFERENCES persons (id)
    LABEL Knows,
  works_at
    SOURCE KEY (person_id) REFERENCES persons (id)
    DESTINATION KEY (company_id) REFERENCES companies (id)
    LABEL WorksAt
);

Under the hood, CREATE PROPERTY GRAPH builds CSR indexes (via Rel::from_edges()) for each edge table and registers the graph metadata in the session catalog.

GRAPH_TABLE — Basic Pattern Matching

Query the graph using GRAPH_TABLE with Cypher-like MATCH patterns:

-- Who does Alice know?
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person WHERE a.name = 'Alice')-[k:Knows]->(b:Person)
  COLUMNS (b.name, b.age)
);
-- Where does Bob work?
SELECT * FROM GRAPH_TABLE (social
  MATCH (p:Person WHERE p.name = 'Bob')-[w:WorksAt]->(c:Company)
  COLUMNS (c.name AS company)
);

Mapping to Engine Ops

The PGQ planner translates MATCH patterns into existing engine operations:

MATCH PatternEngine OpDescription
(a)-[e]->(b) graph.expand() 1-hop neighbor expansion via CSR
(a)-[e]->+(b) graph.var_expand() Variable-length BFS
ANY SHORTEST (a)-[e]->+(b) graph.shortest_path() BFS shortest path
(a)->(b)->(c), (a)->(c) graph.wco_join() Cyclic pattern via Leapfrog Triejoin

Key point: GRAPH_TABLE returns a regular SQL table. This means graph results compose naturally with GROUP BY, ORDER BY, JOIN, subqueries, window functions — everything SQL already does well.

Phase 2: Variable-Length Paths + Shortest Path

TRAVERSAL

Add quantified path patterns and shortest path support to MATCH.

Variable-Length Paths

-- Friends within 1-3 hops
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person WHERE a.name = 'Alice')-[k:Knows]->{1,3}(b:Person)
  COLUMNS (b.name, b.city)
);
-- Reachable in any number of hops
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person WHERE a.id = 42)-[k:Knows]->+(b:Person)
  COLUMNS (b.name)
);

Path quantifiers map directly to var_expand() parameters:

SQL/PGQ QuantifierEngine CallMeaning
-[e]->expand()Exactly 1 hop
-[e]->{2,5}var_expand(min=2, max=5)2 to 5 hops
-[e]->+var_expand(min=1, max=255)1 or more hops
-[e]->*var_expand(min=0, max=255)0 or more hops

Shortest Path

-- Shortest path from Alice to Bob
SELECT * FROM GRAPH_TABLE (social
  MATCH p = ANY SHORTEST
    (a:Person WHERE a.name = 'Alice')-[k:Knows]->+(b:Person WHERE b.name = 'Bob')
  COLUMNS (path_length(p) AS distance)
);
-- Shortest paths from Alice to everyone, with path details
SELECT * FROM GRAPH_TABLE (social
  MATCH p = ANY SHORTEST
    (a:Person WHERE a.name = 'Alice')-[k:Knows]->+(b:Person)
  COLUMNS (b.name, path_length(p) AS hops)
)
ORDER BY hops;

Phase 3: Cyclic Patterns + Graph Analytics

ADVANCED

Triangle Detection

Cyclic patterns like triangles map to wco_join():

-- Find all triangles: three people who all know each other
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person)-[:Knows]->(b:Person)-[:Knows]->(c:Person),
        (a)-[:Knows]->(c)
  COLUMNS (a.name AS person1, b.name AS person2, c.name AS person3)
);

Graph + Full SQL Composability

Because GRAPH_TABLE returns a regular table, graph results compose with the entire SQL language:

-- Top 10 most-connected people (graph + aggregation)
SELECT person, COUNT(*) AS connections
FROM GRAPH_TABLE (social
  MATCH (a:Person)-[:Knows]-(b:Person)
  COLUMNS (a.name AS person)
)
GROUP BY person
ORDER BY connections DESC
LIMIT 10;
-- Average network reach by company (graph + join + aggregation)
SELECT company, AVG(reach) AS avg_reach
FROM GRAPH_TABLE (social
  MATCH (p:Person)-[:WorksAt]->(c:Company),
        (p)-[:Knows]->{1,3}(friend:Person)
  COLUMNS (c.name AS company, COUNT(friend.id) AS reach)
)
GROUP BY company
ORDER BY avg_reach DESC;
-- Graph results as a CTE, then join with other tables
WITH influencers AS (
  SELECT person_id, connections
  FROM GRAPH_TABLE (social
    MATCH (a:Person)-[:Knows]-(b:Person)
    COLUMNS (a.id AS person_id, COUNT(b.id) AS connections)
  )
  WHERE connections > 100
)
SELECT p.name, p.email, i.connections
FROM influencers i
JOIN persons p ON p.id = i.person_id
ORDER BY i.connections DESC;

Graph Algorithms as SQL Functions

-- PageRank (future)
SELECT * FROM GRAPH_TABLE (social
  MATCH (p:Person)
  COLUMNS (p.name, PAGERANK(social, p, 20, 0.85) AS rank)
)
ORDER BY rank DESC LIMIT 10;

-- Connected components (future)
SELECT * FROM GRAPH_TABLE (social
  MATCH (p:Person)
  COLUMNS (p.name, CONNECTED_COMPONENT(social, p) AS component)
);

SQL/PGQ vs Cypher vs Datalog

The same query in three languages — "Find friends-of-friends of Alice who work at Google":

SQL/PGQ (our target)

SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person WHERE a.name = 'Alice')
        -[:Knows]->{2,2}
        (fof:Person)-[:WorksAt]->(c:Company WHERE c.name = 'Google')
  COLUMNS (fof.name, fof.email)
);

Cypher

MATCH (a:Person {name: 'Alice'})-[:KNOWS*2..2]->(fof:Person)-[:WORKS_AT]->(c:Company {name: 'Google'})
RETURN fof.name, fof.email

Similar readability, but no composability with SQL analytics.

Datalog

fof(X, Y) :- knows(X, Z), knows(Z, Y).
result(Name, Email) :- person('Alice', AliceId),
                       fof(AliceId, FofId),
                       works_at(FofId, CompId),
                       company(CompId, 'Google'),
                       person_attrs(FofId, Name, Email).

Elegant for recursion, but aggregation/sorting need non-standard extensions. No SQL tool ecosystem.

End-to-End Example: Social Network

A complete session demonstrating the SQL/PGQ workflow:

-- 1. Load relational data
CREATE TABLE persons AS SELECT * FROM read_csv('persons.csv');
CREATE TABLE companies AS SELECT * FROM read_csv('companies.csv');
CREATE TABLE friendships AS SELECT * FROM read_csv('friendships.csv');
CREATE TABLE employment AS SELECT * FROM read_csv('employment.csv');

-- 2. Define the property graph
CREATE PROPERTY GRAPH social
VERTEX TABLES (
  persons    LABEL Person,
  companies  LABEL Company
)
EDGE TABLES (
  friendships
    SOURCE KEY (person1_id) REFERENCES persons (id)
    DESTINATION KEY (person2_id) REFERENCES persons (id)
    LABEL Knows,
  employment
    SOURCE KEY (person_id) REFERENCES persons (id)
    DESTINATION KEY (company_id) REFERENCES companies (id)
    LABEL WorksAt
);

-- 3. Basic pattern match: who does Alice know?
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person WHERE a.name = 'Alice')-[:Knows]->(b:Person)
  COLUMNS (b.name, b.city)
);

-- 4. Variable-length: all people within 3 hops of Alice
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person WHERE a.name = 'Alice')-[:Knows]->{1,3}(b:Person)
  COLUMNS (b.name, b.city)
);

-- 5. Shortest path
SELECT * FROM GRAPH_TABLE (social
  MATCH p = ANY SHORTEST
    (a:Person WHERE a.name = 'Alice')-[:Knows]->+(b:Person WHERE b.name = 'Zara')
  COLUMNS (path_length(p) AS hops)
);

-- 6. Graph + analytics: most connected people by company
SELECT company, person, connections
FROM (
  SELECT company, person, connections,
         ROW_NUMBER() OVER (PARTITION BY company ORDER BY connections DESC) AS rn
  FROM GRAPH_TABLE (social
    MATCH (p:Person)-[:Knows]-(friend:Person),
          (p)-[:WorksAt]->(c:Company)
    COLUMNS (c.name AS company, p.name AS person, COUNT(friend.id) AS connections)
  )
)
WHERE rn <= 5
ORDER BY company, connections DESC;

-- 7. Find triangles
SELECT * FROM GRAPH_TABLE (social
  MATCH (a:Person)-[:Knows]->(b:Person)-[:Knows]->(c:Person),
        (a)-[:Knows]->(c)
  COLUMNS (a.name AS p1, b.name AS p2, c.name AS p3)
) LIMIT 100;
