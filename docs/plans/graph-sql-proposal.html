PROPOSAL

Graph Queries in SQL

Extending Teide's SQL layer with graph traversal, relationship management, and composable graph+relational analytics — leveraging the engine primitives we already have.

Context: Why Graph?

Graph databases like FalkorDB and Neo4j have proven the value of property graph queries for knowledge graphs, social networks, fraud detection, and AI/RAG pipelines. FalkorDB's core insight is using sparse matrix algebra (via GraphBLAS) to represent adjacency and execute traversals as matrix operations.

Teide already has a comparable foundation: CSR (Compressed Sparse Row) adjacency indexes with forward/reverse directions, integrated into the same morsel-driven columnar executor that handles SQL. The graph primitives exist in the C engine and are exposed via the Rust API — but not yet accessible through SQL.

This proposal adds graph capabilities to the SQL layer without any C engine changes.

What Teide Already Has

ENGINE

CSR Adjacency Indexes

Double-indexed (forward + reverse) compressed sparse row format. O(1) neighbor degree lookup, O(1) range access. Save/load/mmap persistence.

ENGINE

4 Graph Operations

EXPAND (1-hop), VAR_EXPAND (BFS), SHORTEST_PATH, and WCO_JOIN (Leapfrog Triejoin for cyclic patterns like triangle detection).

OPTIMIZER

Graph-Aware Optimization

Sideways Information Passing (SIP) propagates filters backward through traversals. Factorized vectors avoid materializing cross-products.

EXECUTOR

Unified Relational + Graph

Graph ops compose with filter, join, aggregate, sort, window functions in the same DAG. One optimizer, one executor.

Architectural Advantage

Teide's CSR + morsel-driven columnar execution is more practical than pure linear algebra for many workloads:

  • CSR gives O(1) neighbor access — no matrix multiply needed for selective queries where you only need a few neighbors
  • Columnar engine handles analytics natively — most graph databases struggle with aggregation, joins, and window functions
  • Unified optimizer — graph traversals and relational operations share one optimization pipeline (SIP, predicate pushdown, fusion)
  • No impedance mismatch — graph results flow directly into GROUP BY, ORDER BY, JOIN without serialization boundaries

Most graph databases can't do efficient analytics. Most analytical databases can't do graph traversals. Teide does both in one engine.

Phase 1: Graph Ops via SQL

LOW EFFORT · HIGH VALUE

Expose existing graph primitives through SQL syntax. No C engine changes needed.

Relationship DDL

-- Build a CSR index from a foreign-key column
CREATE RELATIONSHIP orders_of
  FROM orders(customer_id) TO customers;

-- Build from an explicit edge table (social network style)
CREATE RELATIONSHIP follows
  FROM edges(src, dst) NODES 1000000;

-- Persist to disk
SAVE RELATIONSHIP follows TO '/data/rels/follows';

-- Reload (or memory-map for zero-copy)
LOAD RELATIONSHIP follows FROM '/data/rels/follows';

-- Drop
DROP RELATIONSHIP follows;

Graph Table Functions

Each graph operation becomes a table-valued function in the FROM clause:

-- 1-hop expansion: who does Alice follow?
SELECT u.name
FROM EXPAND(follows, 'forward') AS e
JOIN users u ON u._rowid = e._dst
WHERE e._src = (SELECT _rowid FROM users WHERE name = 'Alice');
-- Variable-length BFS: all people within 3 hops
SELECT e._end, e._depth
FROM VAR_EXPAND(follows, 'forward', 1, 3) AS e
WHERE e._start = 42;
-- Shortest path between two nodes
SELECT sp._node, sp._depth
FROM SHORTEST_PATH(follows, 42, 99, 10) AS sp;
-- Triangle detection via worst-case optimal join
SELECT t._v0, t._v1, t._v2
FROM WCO_JOIN(follows, follows, follows, 3) AS t;

Composable with Full SQL

The real power: graph results flow seamlessly into standard SQL operations.

-- Graph + aggregation: count followers per user
SELECT u.name, COUNT(*) AS follower_count
FROM EXPAND(follows, 'reverse') AS e
JOIN users u ON u._rowid = e._dst
GROUP BY u.name
ORDER BY follower_count DESC
LIMIT 10;
-- Graph + analytics: average path depth by community
SELECT c.community, AVG(e._depth) AS avg_reach
FROM VAR_EXPAND(follows, 'forward', 1, 5) AS e
JOIN users u ON u._rowid = e._start
JOIN communities c ON c.user_id = u._rowid
GROUP BY c.community;
-- Subquery-driven expansion: expand from VIP users only
SELECT e._dst
FROM EXPAND(follows, 'forward') AS e
WHERE e._src IN (SELECT _rowid FROM users WHERE tier = 'VIP');

Phase 2: OpenCypher Frontend

MEDIUM EFFORT · BIG UNLOCK

Add a Cypher parser that compiles MATCH patterns down to the same graph ops.

-- Cypher query
MATCH (a:Person)-[:KNOWS]->(b:Person)
WHERE a.name = 'Alice'
RETURN b.name, b.age

This compiles to the same DAG as:

SELECT p2.name, p2.age
FROM EXPAND(knows, 'forward') AS e
JOIN persons p1 ON p1._rowid = e._src
JOIN persons p2 ON p2._rowid = e._dst
WHERE p1.name = 'Alice'

Key deliverables:

  • OpenCypher parser (using nom/pest or the opencypher-parser crate)
  • Cypher planner: MATCH pattern → expand/var_expand/wco_join ops
  • Variable-length path syntax: [:KNOWS*1..3] → var_expand(min=1, max=3)
  • Optional: Bolt wire protocol for Neo4j client compatibility

Phase 3: Property Graph Catalog

MEDIUM EFFORT

Formalize the mapping between tables and graph entities:

-- Register a table as a node label
CREATE NODE LABEL Person FROM persons;

-- Register a relationship type with its source and target labels
CREATE EDGE LABEL KNOWS
  FROM Person TO Person
  USING friendships(person_a, person_b);

-- Query the catalog
SHOW NODE LABELS;
SHOW EDGE LABELS;

This adds:

  • Node label registry: which tables represent which node types
  • Edge label registry: name → Rel mapping with source/target labels
  • Schema validation on graph mutations
  • Automatic relationship building when edge data is inserted

Phase 4: Graph Algorithms

INCREMENTAL

Built-in graph algorithms exposed as SQL functions:

-- PageRank
SELECT node_id, PAGERANK(follows, 20, 0.85) AS rank
FROM users
ORDER BY rank DESC LIMIT 10;

-- Connected components
SELECT node_id, CONNECTED_COMPONENT(follows) AS component_id
FROM users;

-- Community detection (Louvain)
SELECT node_id, COMMUNITY(follows) AS community_id
FROM users;

These can be implemented as iterative graph operations using existing engine primitives. Each algorithm becomes a Rust function that builds a computation DAG, executes it in a loop until convergence, and returns the result as a table.

End-to-End Example: Social Network

A complete session demonstrating the Phase 1 SQL extensions:

-- 1. Load data
CREATE TABLE users AS SELECT * FROM read_csv('users.csv');
CREATE TABLE edges AS SELECT * FROM read_csv('follows.csv');

-- 2. Build the relationship index
CREATE RELATIONSHIP follows FROM edges(src, dst) NODES 1000000;

-- 3. Who are the most-followed users?
SELECT u.name, COUNT(*) AS followers
FROM EXPAND(follows, 'reverse') AS e
JOIN users u ON u._rowid = e._dst
GROUP BY u.name
ORDER BY followers DESC
LIMIT 10;

-- 4. Find all people within 3 hops of user 42
SELECT u.name, e._depth
FROM VAR_EXPAND(follows, 'forward', 1, 3) AS e
JOIN users u ON u._rowid = e._end
WHERE e._start = 42
ORDER BY e._depth;

-- 5. Shortest path between two users
SELECT u.name, sp._depth
FROM SHORTEST_PATH(follows, 42, 99, 10) AS sp
JOIN users u ON u._rowid = sp._node;

-- 6. Find triangles (mutual follow clusters)
SELECT u0.name, u1.name, u2.name
FROM WCO_JOIN(follows, follows, follows, 3) AS t
JOIN users u0 ON u0._rowid = t._v0
JOIN users u1 ON u1._rowid = t._v1
JOIN users u2 ON u2._rowid = t._v2
LIMIT 100;

-- 7. Save the index for later
SAVE RELATIONSHIP follows TO '/data/indexes/follows';

Why SQL First, Not Cypher?

  • Zero new parser — sqlparser already handles table functions and custom syntax via the DuckDB dialect
  • Zero engine changes — every graph op already exists and is tested in the C engine
  • Full composability — graph results flow into joins, aggregations, window functions, ORDER BY — things Cypher handles poorly
  • PgWire compatible — any Postgres client (DBeaver, psql, psycopg2) can run graph queries immediately
  • Incremental path — Cypher can be added later as a separate frontend that compiles to the same graph ops
  • SQL/PGQ alignment — the ISO SQL/PGQ standard is adding graph pattern matching to SQL; our table-function approach is a pragmatic step in that direction

Bottom line: The C engine already does the hard work. Phase 1 is ~600 lines of planner code to unlock graph queries for every SQL client.
